# ADR-008: Supply-chain security — SBOM generation and attestation strategy

Status: Accepted
Date: 2026-05-26
Owners: pvandervelde

## Context

Consumers of the release-regent container image have no current way to verify
what software components are inside it, how it was built, or whether the image
they pulled is the same one this repository produced. As supply-chain attacks
become more common, providing verifiable provenance and a software bill of
materials (SBOM) is expected practice for any publicly distributed artefact.

Two distinct SBOM concerns exist for a Rust/Docker project:

1. **Source-level dependencies** — the Cargo crates resolved at build time
   (`Cargo.lock`). These are the transitive dependencies of the application code.
2. **Container-level contents** — every OS package, library, and binary present
   in the final image layers, which includes the compiled Rust binary plus the
   base image contents.

Signing and attestation must use a mechanism that is trustworthy without
requiring maintainers to manage long-lived private keys.

## Decision

For every tag-triggered container release:

- Generate a **Rust source SBOM** (SPDX JSON) from the Cargo workspace using
  Syft (`anchore/sbom-action` scanning `path: .`), which reads `Cargo.lock` to
  enumerate all resolved crate dependencies.
- Generate a **container SBOM** (SPDX JSON) from the published image using Syft
  (`anchore/sbom-action` scanning the pushed image digest), capturing all OS
  packages and binaries in the final image.
- Attach both SBOMs as signed attestations on the container image using
  `actions/attest-sbom`, which writes to the GitHub Attestations store and
  pushes OCI referrers to the registry.
- Generate a **SLSA v1.0 build provenance** attestation using
  `actions/attest-build-provenance`, signed via Sigstore/Fulcio using the
  GitHub Actions OIDC identity of the workflow run (keyless signing — no
  long-lived keys required).
- **Disable BuildKit inline provenance** (`provenance: false`) and **explicitly
  disable BuildKit SBOM generation** (`sbom: false`) on the build step to avoid
  duplicate, unverified records that conflict with the externally generated
  attestations.
- Distribute SBOMs via **OCI referrers and the GitHub Attestations API** only.
  SBOMs are not attached to GitHub Release assets. Workflow artifact copies are
  uploaded as a 90-day convenience cache; they are not the canonical copy.

## Consequences

- Consumers can verify image provenance and inspect the full dependency tree
  without trusting any external key infrastructure.
- Two SBOM types are always co-located with the image in the registry,
  discoverable via standard OCI referrer APIs.
- Attestations are verifiable with:

  ```sh
  gh attestation verify oci://ghcr.io/pvandervelde/release-regent:<version> \
      --owner pvandervelde
  ```

- Users who do not have registry access or OCI tooling cannot directly download
  SBOMs from a GitHub Release page; they must use `gh attestation` or an OCI
  client. This is an accepted trade-off in favour of keeping releases clean and
  making the OCI referrer the single canonical source of truth.
- Workflow artifact copies expire after 90 days. This is intentional; the OCI
  referrers are permanent.

## Alternatives considered

- **CycloneDX format**: CycloneDX is well-supported and richer for VEX
  workflows, but SPDX JSON is the format required by the GitHub Attestations
  API's `actions/attest-sbom` action. SPDX was chosen for compatibility.
- **`cargo-cyclonedx` for source SBOM**: Provides richer Cargo-specific
  metadata (component hashes, purl) but requires installing an additional cargo
  tool in CI. Syft reading `Cargo.lock` produces equivalent component coverage
  and avoids a separate install step, keeping both SBOMs generated by the same
  tool.
- **Cosign + keyless signing separately**: Functionally equivalent to
  `actions/attest-*`, but requires managing `cosign` installation and produces
  attestations outside the GitHub Attestations store (discoverable only via
  Rekor). Using `actions/attest-*` stores attestations both in the GitHub store
  (queryable via `gh attestation`) and as OCI referrers, giving two discovery
  paths.
- **Attach SBOMs to GitHub Releases**: Would make SBOMs accessible without OCI
  tooling, but duplicates distribution and creates a mutable attachment that is
  not cryptographically bound to the image. OCI referrers are the preferred
  supply-chain distribution mechanism.
- **GPG signing**: Requires managing and rotating long-lived private keys.
  Sigstore/Fulcio keyless signing is simpler and equally verifiable via the
  transparency log.

## Implementation notes

- `id-token: write` permission is required for Sigstore/Fulcio OIDC signing.
- `attestations: write` permission is required to write to the GitHub
  Attestations store.
- The `steps.build.outputs.digest` output of `docker/build-push-action`
  (`sha256:…`) is used as the attestation subject for all three attestation
  steps, ensuring every record is pinned to the immutable image digest rather
  than a mutable tag.
